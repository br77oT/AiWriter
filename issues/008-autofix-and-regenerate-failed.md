## What to build

The two recovery actions in the Validation rail per PRD §API contracts → `POST /autofix` and per `prd/make ux wireframes.md` "Validation view" footer buttons.

- **Auto-fix missing items** — regenerates only the sections that contain failing checks (status `missing` or `partial`). Uses the Generation Engine's section-rewrite mode (Slice 007) under the hood, with the failing check's `suggestion` injected as the instruction.
- **Regenerate only failed sections** — regenerates only the outline sections marked `missing` or `thin` by the Structural Evaluator. Same mechanism: section-rewrite per failing outline ID.
- **API**: `POST /autofix { documentId } → DraftSections`. Returns the patched draft sections (only the regenerated ones); the rest of the draft is untouched. Locked sections are skipped (per Slice 007 semantics) — emit a UI notice if a failing item lives in a locked section.
- **UI**: both buttons live in the Validation rail footer per the wireframe. After click, generation runs, the rail re-validates, and the user sees the rail flip statuses to ✓ where successful.

Note: the two buttons share most of their plumbing — the difference is which set of sections is selected (failing-by-question vs. failing-by-structure). Implement them as one shared path with two entry points.

## Acceptance criteria

- [ ] "Auto-fix missing items" regenerates only sections that contain a `missing` or `partial` check; other sections are untouched.
- [ ] "Regenerate only failed sections" regenerates only outline sections marked `missing` or `thin`; other sections are untouched.
- [ ] `POST /autofix` endpoint exists and returns the patched DraftSections.
- [ ] Locked sections are skipped by both flows; the UI surfaces a notice when a failing item is in a locked section.
- [ ] After either action, validation re-runs and the rail updates.
- [ ] Tests: auto-fix only touches failing sections; regenerate-failed-sections only touches failing structural sections; locked sections are never regenerated by either flow.
